# OrderWorder — WhatsApp Templates (Section 7.6)

> **Canonical source of truth for env vars:** `.env.example`.
> **Canonical source of truth for n8n workflows:** `docs/n8n/README.md`.

This document lists the **6 WhatsApp message templates** that the Phase-3 n8n workflows send. Each template must be submitted for approval in the **Meta WhatsApp Manager** before the corresponding workflow can deliver messages. Until a template is **Approved**, the n8n HTTP Request node that references it will receive a `400` from the Meta Graph API with error code `132013` (template not found).

| Status legend | Meaning |
|---|---|
| **To be submitted** | Operator has not yet submitted the template to Meta. |
| **Pending review** | Submitted; Meta review in progress (typically 1–48 hours). |
| **Approved** | Meta approved; the workflow can send messages. |
| **Rejected** | Meta rejected; revise body text and resubmit. |

> All 6 are **UTILITY** category (not MARKETING) — they are transactional notifications tied to a specific customer action (order placed, payment completed, refund issued) or a daily operational summary. UTILITY templates have higher approval rates and faster review than MARKETING templates.

## Meta WhatsApp Manager submission steps (general)

1. Go to **https://business.facebook.com** → select your Business portfolio.
2. Navigate to **WhatsApp Manager** → **Message templates** → **Create template**.
3. Choose:
   - **Category:** UTILITY (for all 6 below)
   - **Name:** exact string from the table (lowercase, underscores)
   - **Language:** English (en) — or add additional languages per market
   - **Body:** copy-paste the EXACT body text from this doc (the `{{1}}`, `{{2}}`, … placeholders must match — Meta rejects if the parameter count differs).
4. Submit for review. Meta typically responds within 1–48 hours.
5. Once **Approved**, the template is available to the n8n workflows via the Meta Graph API `/{phone_number_id}/messages` endpoint with `"type":"template"`.

> **Parameter order matters.** The `{{1}}`, `{{2}}`, … placeholders in the template body MUST match the order of the `parameters` array in the n8n HTTP Request node's `jsonBody`. The mapping is documented per-template below.

## 1. `direct_order_offer`

| Field | Value |
|---|---|
| **Name** | `direct_order_offer` |
| **Category** | UTILITY |
| **Language** | en |
| **Triggered by** | n8n workflow `commission_saver.json` — fired D+3 (3 days after QR scan) to a customer who has NOT placed a 2nd order in that window |
| **Status** | To be submitted |

### Body

```
Hi! Here's your code {{1}} for 10% off your next direct order (no app fees). Tap: {{2}}
```

### Parameters (in order)

| # | Placeholder | Type | Source | Example |
|---|---|---|---|---|
| 1 | `{{1}}` | text | Offer code generated by `/api/internal/customer/eligibility` (e.g. `SAVE10-AB12CD`) | `SAVE10-AB12CD` |
| 2 | `{{2}}` | text | Direct-order URL: `${PUBLIC_URL}/<restaurantID>/direct?code=<offerCode>` | `https://app.orderworder.com/empire/direct?code=SAVE10-AB12CD` |

### Sample submission body (Meta API)

```json
{
  "name": "direct_order_offer",
  "category": "UTILITY",
  "language": "en",
  "components": [
    {
      "type": "BODY",
      "text": "Hi! Here's your code {{1}} for 10% off your next direct order (no app fees). Tap: {{2}}"
    }
  ]
}
```

## 2. `daily_owner_report`

| Field | Value |
|---|---|
| **Name** | `daily_owner_report` |
| **Category** | UTILITY |
| **Language** | en |
| **Triggered by** | n8n workflow `owner_daily_report.json` — fired at 11 PM IST (19:00 UTC) daily for every active restaurant |
| **Status** | To be submitted |

### Body

```
📋 {{1}} Daily Report
Bills: {{2}} | Sale: ₹{{3}} | GST: ₹{{4}}
Low-stock items: {{5}} | Theft alerts: {{6}}
Shift variance: ₹{{7}}
— OrderWorder
```

### Parameters (in order)

| # | Placeholder | Type | Source | Example |
|---|---|---|---|---|
| 1 | `{{1}}` | text | Date (ISO date string) returned by `/api/internal/daily-summary` | `2026-07-24` |
| 2 | `{{2}}` | text | Total bills count | `47` |
| 3 | `{{3}}` | text | Net sales in INR (no thousands separator) | `28450` |
| 4 | `{{4}}` | text | Total GST collected (CGST + SGST + IGST) in INR | `5121` |
| 5 | `{{5}}` | text | Count of low-stock items (integer) | `3` |
| 6 | `{{6}}` | text | Count of theft-suspected inventory items | `0` |
| 7 | `{{7}}` | text | Shift variance in INR (signed: `+`/`-` or `0`) | `-150` |

### Sample submission body

```json
{
  "name": "daily_owner_report",
  "category": "UTILITY",
  "language": "en",
  "components": [
    {
      "type": "BODY",
      "text": "📋 {{1}} Daily Report\nBills: {{2}} | Sale: ₹{{3}} | GST: ₹{{4}}\nLow-stock items: {{5}} | Theft alerts: {{6}}\nShift variance: ₹{{7}}\n— OrderWorder"
    }
  ]
}
```

> **Note on emojis:** Meta generally accepts emoji in UTILITY templates. If your submission is rejected for "invalid character," replace `📋` with `Daily Report:` and re-submit.

## Testing templates (sandbox)

Before submitting to Meta, you can test message rendering locally:

1. Set `WHATSAPP_PHONE_NUMBER_ID` + `WHATSAPP_ACCESS_TOKEN` in `.env.local` (Meta test number from https://developers.facebook.com/apps/ → your app → WhatsApp → API Setup).
2. Add your test phone number to the Meta app's "To" allow-list (test numbers can only send to allow-listed recipients).
3. Hit `/api/whatsapp/send` directly with a test payload:

   ```bash
   curl -X POST http://localhost:3050/api/whatsapp/send \
     -H "Content-Type: application/json" \
     -d '{
       "to": "+919876543210",
       "template": "rate_your_visit",
       "parameters": ["Test Restaurant", "https://example.com/feedback/test-token"]
     }'
   ```

4. Verify the message arrives on the test phone.

> Templates MUST be **Approved** in Meta WhatsApp Manager before they can be sent — even in the test environment. The Meta test phone number has 3 pre-approved sample templates (`hello_world`); your custom templates go through the same review pipeline.

## Post-approval checklist

- [ ] All 6 templates show **Approved** in Meta WhatsApp Manager.
- [ ] Each template's parameter order matches the corresponding n8n workflow's `jsonBody.parameters` array (verify by opening the workflow in n8n and inspecting the HTTP Request node).
- [ ] `WHATSAPP_PHONE_NUMBER_ID` and `WHATSAPP_ACCESS_TOKEN` are set in both the app's `.env` and n8n's environment variables.
- [ ] Test-fire each of the 5 n8n workflows (see `docs/n8n/README.md` "Test-fire steps") and confirm the WhatsApp message arrives on a real device.
- [ ] Update the Status column at the top of this doc from "To be submitted" → "Approved" for each template as it lands.
